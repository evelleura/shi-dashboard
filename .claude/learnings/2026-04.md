## 2026-04-04 | ACC | Score: 16 | CLAUDE.md spec diverges significantly from actual user stories

**What happened:** User stories in `docs/user-story/user-stories.md` reveal requirements far broader than CLAUDE.md captures. Actual needs include: task-level tracking (not just daily progress %), survey phase before execution, evidence uploads (photos/docs/screenshots), Kanban/table views for technicians, manager budget planning, per-task technician assignment with auto-assign, client management (name/address/phone), and project value in rupiah. Current DB schema (`daily_reports` with `progress_percentage`) and SPI calculator are built against the simpler CLAUDE.md model.
**Why it matters:** Every feature built against CLAUDE.md will need rework once actual user stories are implemented. Data model mismatch is the most expensive kind of technical debt.
**Action for next time:** Before any implementation phase, reconcile CLAUDE.md spec with user stories document. Update data model to match actual requirements FIRST. Flag any spec-vs-stories conflicts to user immediately.
**Files involved:** `CLAUDE.md`, `docs/user-story/user-stories.md`, `server/database/schema.sql`, `server/src/services/spiCalculator.ts`

## 2026-04-04 | FIX | Score: 19 | Manual daily reports are an anti-pattern -- task board IS the report

**What happened:** Built a DailyReportForm where technicians manually type progress percentages. User rejected it with visible frustration: "not some bullshit that I can fill the percentage myself." The reference app (task15-daily-task-planner) confirms: task status changes ARE the daily activity log. Manual percentage entry is redundant when tasks already track completion. SPI already calculates from task completion ratio. Removed: ReportPage, DailyReportForm, useReports hook, submitReport API, DailyReport type, Reports tab from ProjectDetailPage.
**Why it matters:** Manual progress percentage is a thesis-era metric that doesn't match how real technicians work. They update task statuses; the system derives progress. Building a manual report form on top of task tracking is asking users to report the same information twice.
**Action for next time:** Before building any "report" feature, ask: "Is this data already captured by task status changes?" If yes, auto-derive it. Never ask users to manually enter what the system already knows. The task board IS the daily report.
**Files involved:** `frontend/src/pages/ReportPage.tsx` (deleted), `frontend/src/components/forms/DailyReportForm.tsx` (deleted), `frontend/src/services/api.ts`, `frontend/src/hooks/useDashboard.ts`, `frontend/src/types/index.ts`

## 2026-04-04 | FIX | Score: 16 | Technician cannot mark "done" -- only manager reviews and approves completion

**What happened:** User clarified role boundaries: technicians set tasks to to_do/working_on_it but CANNOT mark "done". Only managers can approve completion after reviewing evidence. Also: technicians need to see client name + address (they go on-site), project list, and evidence upload per task. Previous implementation gave technicians full status control and a Daily Report page instead of projects/clients views. Redesigned: TaskStatusSelect accepts userRole prop, technician sidebar = Dashboard/MyProjects/MyTasks, task detail modal includes evidence upload tab.
**Why it matters:** Role permissions aren't just about access control -- they model the actual business workflow. Technician = field worker who reports progress. Manager = reviewer who approves. Giving technicians "done" permission bypasses the review gate that the business process requires.
**Action for next time:** Map real-world workflow before implementing role permissions. Ask: "Who does the work?" vs "Who approves the work?" These are always different roles. Status transitions should reflect approval chains, not just task ownership.
**Files involved:** `frontend/src/components/tasks/TaskStatusSelect.tsx`, `frontend/src/pages/TechnicianDashboard.tsx`, `frontend/src/components/ui/Layout.tsx`, `frontend/src/App.tsx`, `server/src/routes/tasks.ts`

## 2026-04-04 | FIX | Score: 14 | Computed states (overtime/over_deadline) beat manual states (stuck)

**What happened:** User rejected manual "stuck" status. Replaced with auto-computed categories: Overtime (status=working_on_it AND due_date < today) and Over Deadline (status=to_do AND due_date < today). DB statuses reduced from 4 to 3 (to_do, working_on_it, done). Kanban shows 5 visual columns with the extra 2 computed from date comparison. Changed schema, 7 backend files, 17+ frontend files, all seed data.
**Why it matters:** "Stuck" requires technician self-reporting a problem. Overtime/over_deadline are objective facts derived from dates. Computed states can't be gamed, forgotten, or require user action to trigger. They surface problems automatically.
**Action for next time:** Before adding a manual status, ask: "Can the system compute this from existing data?" If due_date + current_status already tells you the task is in trouble, don't add a manual flag for it. Prefer computed indicators over self-reported states.
**Files involved:** `server/database/schema.sql`, `server/src/routes/tasks.ts`, `server/src/routes/dashboard.ts`, `server/src/services/spiCalculator.ts`, `frontend/src/components/tasks/KanbanBoard.tsx`

## 2026-04-04 | FIX | Score: 13 | Dashboard API endpoints must return full entity data for detail modals

**What happened:** Technician task Kanban showed "Assigned: Unassigned" and "Budget: --" because the dashboard endpoint only selected 5 columns (id, name, status, due_date, project_name) instead of `t.*` with JOINs. The TaskDetailModal expected assigned_to_name, budget, description, evidence_count -- all missing. Also had LIMIT 10 which silently cut off tasks. Fixed by selecting `t.*` with LEFT JOINs for user name and evidence count, removed LIMIT.
**Why it matters:** Dashboard summary endpoints are often optimized for list views but then reused for detail modals. If the detail modal needs more fields than the list card, the API must return them or the modal shows blanks.
**Action for next time:** When a list view has a "click for details" modal, the list API must return ALL fields the modal needs. Check modal component props against API SELECT columns before shipping. Or fetch full entity on modal open via separate endpoint.
**Files involved:** `server/src/routes/dashboard.ts` (technician endpoint), `frontend/src/components/tasks/TaskDetailModal.tsx`

## 2026-04-04 | ACC | Score: 12 | Portable PostgreSQL auto-install eliminates Docker dependency for dev

**What happened:** User rejected Docker for database. run.py now auto-downloads PostgreSQL 17.5 portable binaries (~300MB from get.enterprisedb.com), extracts to `postgres/` directory, runs initdb, starts via pg_ctl, creates database. Subsequent runs detect existing install and just start. Ctrl+C cleanly stops frontend + backend + postgres. Also added native binding detection for Vite/rolldown (auto-reinstall if broken).
**Why it matters:** Zero-dependency development setup. New developer runs `.\run.ps1` and everything works -- no Docker, no system PostgreSQL install, no manual steps.
**Action for next time:** For projects needing PostgreSQL: use portable binaries approach. Download URL pattern: `https://get.enterprisedb.com/postgresql/postgresql-{version}-windows-x64-binaries.zip`. Extract `pgsql/` contents, initdb with `-A trust`, pg_ctl start. Add `postgres/` to .gitignore.
**Files involved:** `run.py`, `run.ps1`, `.gitignore`
