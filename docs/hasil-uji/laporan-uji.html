2 Bug / catatan yang ditemukan

B1 · Judul grafik "Projects by Category" masih Bahasa Inggris RINGAN

Di Dashboard manajer, grafik lain Bahasa Indonesia (Distribusi Kesehatan Proyek, Tugas per Status, Beban Kerja Teknisi), TAPI satu grafik tetap Inggris: "Projects by Category" + teks kosong "No project data available". Kelihatan saat sidang.

Lokasi: frontend/src/components/charts/ProjectCategoryPieChart.tsx baris 34/53/54/60 (hardcode, tidak pakai i18n seperti chart lain).

Fix: ganti judul jadi id ? 'Proyek per Kategori' : 'Projects by Category' & teks kosong jadi 'Belum ada data proyek'. (1-2 baris. Aku bisa kerjakan kalau kamu mau.)

B2 · Dockerfile rusak: COPY next.config.mjs padahal file next.config.ts MENENGAH

Tahap akhir Dockerfile menyalin next.config.mjs yang tidak ada (project pakai .ts). Artinya python run.py --build / build image produksi akan GAGAL. Dev (--new) tidak terpengaruh.

Lokasi: frontend/Dockerfile baris COPY --from=builder /app/next.config.mjs ./.

Fix: ganti .mjs → .ts (atau hapus baris itu kalau tak dibutuhkan saat runtime).

B3 · Peta klien butuh internet (OpenStreetMap) RISIKO DEMO

Preview peta pakai tile OpenStreetMap online. Saat uji (headless tanpa internet) tile gagal → peta abu-abu kosong (koordinat tetap terdeteksi). Kalau ruang sidang tidak ada internet, peta akan kosong.

Antisipasi: pastikan ada internet saat demo klien, ATAU saat sidang cukup tunjukkan "Koordinat terdeteksi" (tanpa fokus ke peta). Bukan crash.

B4 · Dashboard tidak update REAL-TIME kalau diam / lintas-sesi MENENGAH

Ini kemungkinan "data ga auto refresh" yang kamu alami. Faktanya:

  • Update OK saat kamu pindah halaman ke Dashboard (terbukti 14=15).
  • TAPI kalau kamu diam menatap Dashboard lalu data berubah di tempat lain, layar tidak ikut berubah sampai auto-poll 5 menit atau kamu F5. (Tidak ada push real-time / websocket.)
  • Begitu juga 2 jendela browser (manajer + teknisi): perubahan di satu jendela tidak langsung muncul di jendela lain tanpa refresh.

Sumber: useDashboard (frontend/src/hooks/useDashboard.ts) pakai refetchInterval 5 menit; mutasi tugas (useTasks.ts) invalidate ['dashboard'] tapi hanya berlaku saat query di-mount ulang. Grafik ['charts',...] tidak di-invalidate mutasi — menyegar saat remount/stale-2-menit.

Sesuai catatan naskah TC-MN-04 ("indikator EWS tidak update real-time setelah review gate; butuh refresh manual"). Guide sudah menyarankan F5. Kalau mau benar real-time: tambah polling pendek / invalidate ['charts'] juga / websocket (di luar scope sidang).

B5 · SPI "meledak" di proyek yang baru mulai CATATAN METRIK

Proyek yang baru dibuat tampil Critical/merah seketika (SPI 0, karena 0 tugas selesai). Begitu 1 tugas diselesaikan di hari pertama, SPI melonjak ke 10.000 (nilai maksimum/cap) & jadi On Track/hijau. Sebabnya: di awal proyek, "waktu berjalan" (PV) sangat kecil, jadi rasio EV/PV jadi besar sekali.

Bukan crash — perilaku rumus yang benar. Tapi penguji bisa bertanya "kenapa proyek 1 hari sudah 10.000 hijau?". Siapkan jawaban (lihat guide Bagian 5/8), atau pakai proyek dengan rentang tanggal wajar saat demo RAG.

B6 · (Info) Mode --iso menambah baris di tsconfig.json INFO

Saat --iso jalan, Next.js otomatis menambah .next-iso/types/**/*.ts ke frontend/tsconfig.json include. Tidak berbahaya (cuma glob types). Bisa dirapikan manual kalau mengganggu.

3 Cara ulang uji ini (reusable)

Semua sudah jadi code, tinggal jalankan:

Langkah Perintah
1. Nyalakan stack terisolasi (DB 5544 + web 3100, empty) python run.py --new --iso
2. Jalankan walkthrough + screenshot cd docs/hasil-uji && python walkthrough.py
3. Hentikan + bersihkan stack iso python run.py --iso --stop (lalu Ctrl+C di terminal dev)
Catatan Stack iso tidak menyentuh stack utamamu (:3000 / DB 5433). Jadi kamu bisa lanjut kerja di :3000 sambil aku uji di :3100. Screenshot tiap run ada di folder ini (docs/hasil-uji/): 01..16 + _console-errors.json.

Cara baca (korelasi antar log)

Mulai dari actions-timeline.log. Tiap baris ber-timestamp jam:menit:detik, jadi bisa dicocokkan ke next-dev.log (server+API) dan postgres.log (query DB) di jam yang sama. Contoh nyata satu aksi: 

==============================================================================
STEP 02 | 2026-06-18 13:41:34 | Buat klien + Link Google Maps
------------------------------------------------------------------------------
  13:41:34  [api] 200 GET    /api/clients          ← muat daftar klien
  13:41:39  [shot] 04-klien-form-terisi-maps.png
  13:41:39  [api] 201 POST   /api/clients          ← SIMPAN klien (sukses)
  13:41:39  [api] 200 GET    /api/clients          ← daftar di-refresh
  13:41:41  [shot] 05-klien-tersimpan.png
Temuan dari log Log menangkap 409 Conflict saat membuat tugas ke-2 (jadwal overlap dengan tugas ke-1). Itu BUKAN bug — fitur anti-dobel-booking bekerja (teknisi tak boleh 2 tugas di rentang waktu sama). Untuk buat 2 tugas, beri rentang jadwal berbeda.
Cara regen + bekukan untuk handoff python run.py --new --iso (nyalakan, mulai rekam) → cd docs/hasil-uji && python walkthrough.py (isi timeline/api/console) → python run.py --iso --stop (bekukan). Lalu zip folder iso-logs/ + docs/hasil-uji/ untuk diserahkan.
