Kajian ini meninjau beberapa penelitian terdahulu yang relevan dengan 
pengembangan fitur dashboard pada sistem manajemen projek. Literatur tersebut

Penelitian Azkia dkk. (2024) mengatasi kendala keterlambatan dan 
pembengkakan anggaran proyek melalui perancangan dashboard monitoring and 
controlling menggunakan metode Agile and Lean Development serta konsep Earned 
Value Management (EVM). Dikembangkan dengan Google Looker Studio dan Google 
Sheets, dashboard ini memuat tujuh fitur utama (Project Overview, Financial Tracking, 
Timeline, Milestone, Resource, Performance, dan Risk). Hasil Black Box Testing 
menunjukkan seluruh fitur berfungsi baik, dan User Acceptance Test (UAT) meraih

bahwa dashboard manajemen mampu meningkatkan akurasi data, mempercepat

2.2 Landasan Teori 
Dalam pengerjaan penelitian Tugas Akhir ini penulis menggunakan beberapa 
landasan teori yang menjadi acuan diantaranya sebagai berikut: 
2.2.1 Manajemen Proyek 
Manajemen proyek menerapkan metode sistematis untuk merencanakan, 
mengeksekusi, dan mengendalikan aktivitas demi mencapai target sesuai batasan 
waktu, anggaran, dan ruang lingkup. Pada fase pengendalian, manajer membutuhkan 
pasokan data akurat secara waktu nyata sebagai dasar evaluasi kinerja dan keputusan 
strategis. Praktik konvensional dengan instrumen tak terintegrasi, seperti lembar kerja 
terpisah, sering memicu kemacetan informasi dan fragmentasi data penugasan. 
Mengatasi kendala tersebut, implementasi sistem manajemen proyek terpadu menjadi 
sangat krusial untuk mengonsolidasikan seluruh kegiatan operasional (Reddy dkk., 
2025).  
2.2.2 Dashboard Manajemen Proyek  
Dalam sistem manajemen proyek, dashboard berfungsi sebagai antarmuka 
pengguna grafis yang memvisualisasikan data operasional secara terpusat. Berbeda 
dengan sistem analitik prediktif, dashboard memfasilitasi transparansi informasi 
dengan mengubah rekapitulasi data laporan proyek tekstual yang diinput oleh teknisi 
menjadi representasi visual terstruktur, seperti grafik kemajuan fisik dan indikator 
ketepatan waktu terhadap rencana awal atau baseline (Silmina & Azmi, 2025).  
2.2.3 Sistem Informasi Laporan Harian Proyek 
Sistem Informasi Laporan Harian Proyek merupakan modul esensial dalam 
ekosistem Sistem Manajemen Proyek yang berfungsi merekam aktivitas operasional 
harian—seperti progres pekerjaan, alokasi waktu, dan kendala teknis lapangan—secara 
terstruktur. Integrasi pelaporan ini bertujuan untuk menyentralisasi data historis agar 
pelacakan proyek selalu aktual dan terhindar dari risiko ketidakkonsistenan pencatatan 
manual (Alawiyah dkk., 2022).

2.2.4 Status Kesehatan Proyek 
Status kesehatan proyek (Project Health Status) merupakan indikator evaluatif 
faktual yang membandingkan progres aktual di lapangan dengan rencana awal. Alih-
alih menggunakan analitik prediktif yang kompleks, indikator ini berfungsi sebagai 
instrumen audit visual untuk mendeteksi penyimpangan jadwal sedini mungkin. Pada 
implementasi dashboard, status kesehatan direpresentasikan melalui metode Red-
Amber-Green (RAG) yang secara otomatis mengagregasi data mentah dari laporan 
harian. Melalui integrasi metode RAG, nilai SPI yang dihasilkan sistem dikonversi 
menjadi indikator visual berdasarkan ambang batas tertentu, yaitu hijau untuk nilai SPI 
≥ 0,95, kuning (amber) untuk nilai SPI ≥ 0,85, dan merah untuk nilai SPI < 0,85. 
Visualisasi ini berfungsi sebagai EWS yang membantu manajer memantau kondisi 
proyek secara cepat dan objektif tanpa perlu menelusuri laporan harian secara manual 
(Fonseca dkk., 2025). 
2.2.5 Early Warning System Berbasis Schedule Performance Index 
Early Warning System (EWS) merupakan fitur yang diintegrasikan ke dalam 
sistem pemantauan untuk mendeteksi penyimpangan jadwal operasional secara 
otomatis. Dalam konteks sistem informasi manajemen, EWS bekerja dengan 
memproses data progres aktual secara terkomputasi untuk menghasilkan indikator 
peringatan sesuai dengan parameter metrik yang telah ditetapkan. Tujuan utama 
implementasi EWS adalah mempercepat deteksi keterlambatan proyek, sehingga 
manajer dapat segera melakukan tindakan mitigasi yang terarah berdasarkan data 
faktual. 
Sebagai parameter pemicu (trigger), EWS pada sistem ini mengadopsi metrik 
Schedule Performance Index (SPI). SPI bertindak sebagai indikator matematis pasti 
yang dihitung menggunakan rasio perbandingan antara persentase pekerjaan aktual 
yang telah dilaporkan selesai Earned Value (EV) dengan persentase target rencana 
awal proyek (Planned Value). Secara matematis, SPI dirumuskan pada Persamaan 1 
berikut:

2.2.6 Framework Next.js 
Next.js merupakan kerangka kerja (framework) front-end berbasis React yang 
dirancang untuk membangun antarmuka web berskala enterprise dengan performa 
tinggi. Keunggulan utamanya terletak pada dukungan Server-Side Rendering (SSR) 
dan arsitektur berbasis komponen, yang sangat ideal untuk memvisualisasikan elemen 
dashboard secara modular, seperti grafik progres kerja dan indikator warna RAG 
Dalam implementasi EWS pada manajemen proyek, Next.js berperan krusial melalui 
mekanisme penarikan data asinkron. Framework ini mampu mengonsumsi akumulasi 
data laporan proyek dan hasil kalkulasi SPI  dari backend secara real-time. Ketika 
terdeteksi perubahan status proyek menjadi kritis (misalnya 𝑆𝑃𝐼 <  1), antarmuka 
dashboard dapat merefleksikan peringatan visual tersebut secara instan tanpa perlu 
memuat ulang keseluruhan halaman, sehingga manajer dapat memperoleh informasi 
operasional yang faktual secara cepat dan interaktif (Genne, 2025).

2.2.9 Basis Data PostgreSQL 
Basis data (database) merupakan sekumpulan data terstruktur yang berelasi 
secara logis dan disimpan secara terpusat pada media elektronik. Pengorganisasian ini 
dilakukan untuk mempermudah kontrol dan pengelolaan informasi. Sistem ini 
bertindak sebagai arsitektur fundamental untuk menyimpan, memanipulasi, dan 
mengekstraksi informasi dengan mempertahankan integritas serta meminimalisir 
redundansi data (Mahmoud & Yasin, 2025).  
PostgreSQL merupakan sistem manajemen basis data relasional objek 
(ORDBMS) sumber terbuka yang dirancang untuk mengelola data terstruktur 
menggunakan bahasa SQL. Dalam pengembangan sistem ini, PostgreSQL berperan 
sebagai infrastruktur penyimpanan utama yang mengelola relasi antara rekaman proyek 
dan pelaporan harian, sehingga seluruh data mentah tersentralisasi secara konsisten 
sebelum diolah menjadi metrik analitik pada fitur dashboard (Silmina & Azmi, 2025).

2.2.10 Structure Query Language 
Structured Query Language (SQL) merupakan bahasa pemrograman standar 
yang dirancang secara spesifik untuk mengelola, memanipulasi, dan mengekstraksi 
informasi dari Sistem Manajemen Basis Data Relasional (RDBMS). Secara fungsional, 
SQL memfasilitasi interaksi tingkat tinggi antara aplikasi perangkat lunak dan 
repositori data melalui eksekusi kueri (query) terstruktur. Instruksi dalam SQL 
umumnya diklasifikasikan ke dalam beberapa kategori utama, di antaranya Data 
Definition Language (DDL) untuk merancang skema atau struktur tabel, serta Data 
Manipulation Language (DML) seperti perintah SELECT, INSERT, UPDATE, dan DELETE 
yang bertugas memproses data operasional. Keunggulan utama SQL terletak pada 
kemampuannya mengeksekusi fungsi agregasi komputasional, memfilter informasi 
spesifik, serta mengonsolidasikan entitas dari berbagai tabel yang saling berelasi 
melalui operasi JOIN, sehingga memungkinkan pengambilan maupun pengolahan data 
berskala besar secara cepat, akurat, dan terpusat di tingkat database (Khan dkk., 2023). 
2.2.11 Black Box Testing 
Pengujian perangkat lunak dengan metode Black Box berfokus pada evaluasi 
fungsionalitas antarmuka tanpa menelusuri logika internal sistem. Proses ini 
memvalidasi kesesuaian parameter input dengan output, umumnya memanfaatkan 
teknik Equivalence Partitioning guna memetakan domain data valid dan tidak valid. 
Pada pengembangan dashboard ini, metode tersebut diimplementasikan untuk 
memvalidasi operasional front-end dan fungsionalitas API pada framework Next.js. 
Pengujian dititikberatkan pada skenario penginputan laporan harian teknisi, manipulasi 
status tugas, serta keakuratan kalkulasi metrik SPI. Pendekatan ini esensial untuk 
menjamin algoritma EWS dan otorisasi berbasis peran Role-Based Access Control 
(RBAC) berjalan akurat tanpa cacat fungsional sebelum diimplementasikan (Kinasih 
dkk., 2026).
