Rencana produk untuk 4 fitur: (1) kategori eskalasi yang setiap kategorinya benar-benar mengeksekusi tindakan (mis. ganti teknisi -> reassign + reschedule + rekalkulasi seluruh relasi), (2) komentar per proyek untuk komunikasi tim, (3) SPI per-teknisi, dan (4) visualisasinya. Disusun setelah pembacaan langsung kode frontend/ dan graph naskah TA.

2Fitur 1 - Kategori Eskalasi yang Bertindakan

Setiap kategori eskalasi harus mengeksekusi tindakan nyata saat manajer menyetujui, dan tindakan itu meriak ke seluruh relasi terkait (jadwal, penugasan, SPI, notifikasi, audit).

2.1 Keadaan saat ini (terverifikasi di kode)

Di escalations.ts: requestAction (baris 361) menyimpan permintaan (action_request + note, status -> pending). respondActionRequest (baris 407) hanya mem-flip action_request_status jadi approved/rejected.

Celah inti

Satu-satunya efek samping nyata: batalkan_eskalasi yang disetujui -> status='closed'. Persetujuan ganti_teknisi, perpanjang_deadline, ganti_alat, mediasi_client tidak melakukan apa pun ke tb_tugas, jadwal, atau SPI. Manajer harus mengerjakan reassign secara manual di tempat lain. Inilah yang diminta user untuk ditutup.

2.2 Grounding naskah

Naskah TA mendefinisikan eskalasi hanya sampai: priority (low/medium/high), alur open -> handled -> closed, dan "manajer mengirim instruksi penanganan" (metode kelas sendInstruction). Naskah tidak mendefinisikan kategori-ke-tindakan. Sistem tindakan ini adalah pengembangan (terinspirasi MDI/PDI Luthan 2023). Secara akademik, bingkai fitur ini sebagai operasionalisasi konkret sendInstruction: instruksi manajer bukan sekadar teks, tapi tindakan tereksekusi.

Untuk sidang

Jika butuh fidelity naskah ketat, tandai bab ini sebagai "penyempurnaan di luar lingkup minimum naskah". Defensible: naskah memang menyebut sendInstruction; kita menjadikannya eksekutif.

2.3 Katalog kategori -> tindakan

Dua tingkat. Eksekutif = mengubah state & cascade. Tercatat = log + notifikasi (tak ada entitas untuk dimutasi -- naskah membuang materials & tak punya entitas komunikasi-klien).

2.4 Cascade ganti_teknisi (andalan)

User menekankan: "ganti teknisi -> jadwal teknisi & semua relasi yang berhubungan dengan teknisi tsb pasti pengaruh." Berikut peta dampak lengkap (hasil telusur kode), dieksekusi dalam satu transaksi (getClient() + BEGIN/COMMIT/ROLLBACK) supaya gagal-sebagian = rollback.

Langkah transaksi (X = teknisi lama, Y = baru, P = proyek)

1. Validasi: Y aktif & ber-peran teknisi, Y != X.
2. Tentukan himpunan tugas: tugas yang dieskalasi (default) atau semua tugas X di P yang belum done (jika reassign_scope=all).
3. Cek konflik jadwal Y: untuk tiap tugas, findConflicts(Y, timeline_start, timeline_end, taskId) -> bentrok = tolak (409) atau peringatkan.
4. UPDATE tb_tugas SET assigned_to=Y, updated_at=NOW() WHERE id_tugas IN (...).
5. INSERT INTO tb_penugasan_proyek (id_proyek,id_user) VALUES (P,Y) ON CONFLICT DO NOTHING. Jika X tak punya sisa tugas di P -> opsional hapus penugasan X.
6. Notifikasi: Y ("ditugaskan"), X ("dilepas dari tugas"), pelapor ("permintaan disetujui").
7. recalculateSPI(P) -- refresh project_health.
8. logChange() per tugas (assigned_to: X->Y) ke audit_log.
9. Eskalasi: action_request_status='approved', append escalation_updates berisi ringkasan tindakan; opsional status='handled'.
10. COMMIT. (SPI teknisi X & Y tak perlu disentuh -- dihitung live saat query berikutnya.)

Prinsip

Reassign mengubah kepemilikan ke depan (siapa kerjakan sisanya), bukan menulis ulang sejarah (bukti/laporan tetap atas nama X). Itu benar untuk audit & SPI: kontribusi done X tetap miliknya; Y dinilai dari yang ia selesaikan.

2.5 perpanjang_deadline

  • Tugas saja: UPDATE tb_tugas SET due_date=, timeline_end=. Memengaruhi hitungan overdue/overtime, bukan PV proyek langsung.
  • Cascade ke proyek (jika cascade_project_end=true): UPDATE tb_proyek SET end_date=. duration auto (kolom generated). Ini menurunkan PV -> menaikkan SPI seluruh proyek (proyek jadi "lebih tidak terlambat"). Surface efek ini ke manajer sebelum konfirmasi.
  • recalculateSPI(P), audit, notifikasi pelaksana+pelapor, catatan thread. Transaksi tunggal.

2.6 Tindakan tercatat (ganti_alat, mediasi_client)

Tanpa entitas untuk dimutasi, tindakan = (a) tulis escalation_updates ringkasan keputusan, (b) set resolution_notes/lanjutkan status, (c) notifikasi. ganti_alat boleh bundel extra_days kecil (memakai jalur perpanjang_deadline). Tetap minimal -- jangan bikin tabel alat (langgar lingkup naskah).

Sketsa executor (referensi implementasi)

// lib/db.ts -- helper transaksi tipis di atas getClient() yang sudah ada
export async function withTransaction<T>(fn: (c: PoolClient) => Promise<T>): Promise<T> {
  const c = await getClient();
  try { await c.query('BEGIN'); const r = await fn(c); await c.query('COMMIT'); return r; }
  catch (e) { await c.query('ROLLBACK'); throw e; }   // gagal mana pun -> batal semua
  finally { c.release(); }
}

// lib/handlers/escalationActions.ts -- inti ganti_teknisi (dipanggil saat approve)
async function execGantiTeknisi(esc, params, c /* client tx */) {
  const Y = params.new_technician_id;
  const taskIds = params.reassign_scope === 'all'
    ? await openTaskIdsOfTechInProject(esc.reported_by, esc.project_id, c) // semua tugas X belum-done
    : [esc.task_id];                                                       // default: tugas eskalasi
  await assertTechnicianValid(Y, esc.reported_by, c);                      // Y aktif & Y != X
  for (const id of taskIds) await assertNoScheduleConflict(Y, id, c);      // findConflicts -> throw = ROLLBACK
  await c.query(`UPDATE tb_tugas SET assigned_to=$1, updated_at=NOW() WHERE id_tugas = ANY($2)`, [Y, taskIds]);
  await c.query(`INSERT INTO tb_penugasan_proyek (id_proyek,id_user) VALUES ($1,$2) ON CONFLICT DO NOTHING`, [esc.project_id, Y]);
  await recalculateSPI(esc.project_id);          // SPI proyek; SPI teknisi X & Y ikut (live, tak disimpan)
  await notifyReassign(esc, Y, taskIds, c);      // notif X, Y, pelapor
  await logReassign(esc, Y, taskIds, c);         // audit_log per tugas + escalation_updates
}

2.8 Kriteria penerimaan

  • Menyetujui ganti_teknisi dengan Y benar-benar mengubah tb_tugas.assigned_to pada tugas yang ditarget, dan Y muncul di tb_penugasan_proyek.
  • Bentrok jadwal Y diblok/diperingatkan via findConflicts.
  • SPI proyek & SPI kedua teknisi mencerminkan perubahan tanpa langkah manual.
  • Notifikasi terkirim ke X, Y, pelapor; audit_log mencatat tiap perubahan assigned_to.
  • Gagal di tengah (mis. Y tak valid) -> rollback penuh, tak ada perubahan parsial.
  • perpanjang_deadline menggeser tanggal & merekalkulasi SPI; cascade ke end_date hanya bila diminta.
  • Tindakan tercatat menulis thread + notifikasi tanpa membuat tabel baru.

4Fitur 3 - SPI Teknisi

Sudah diimplementasi. PRD ini mendokumentasikannya & menutup risiko drift.

4.1 Rumus (terverifikasi, spiCalculator.ts:78)

SPI_teknisi = earned / planned        // di-cap SPI_CAP=2; RAG via categorizeHealth
  earned  = jumlah tugas teknisi berstatus 'done'
  planned = Sigma(project_health.planned_progress / 100) atas tugas teknisi
            di proyek AKTIF dengan PV >= MIN_PV_FOR_SPI (5%)
  planned <= 0  ->  null ("Belum Dinilai")

Ambang RAG identik proyek: hijau >=0.95, kuning 0.85-0.95, merah <0.85. Dihitung live (tak disimpan) di 3 titik: dashboard teknisi (my_spi), listTechnicians, getTechnicianDetail.

4.2 Grounding naskah

Naskah menyebut "rasio SPI individu" (Gambar 4.23, Dashboard Performa Teknisi) tanpa rumus. Implementasi saat ini adalah operasionalisasi konkretnya -- catat di bab metodologi untuk pertahanan sidang.

4.3 Penyempurnaan (ringan)

  • Anti-drift: ekstrak SQL earned/planned ke satu helper (mis. technicianSpiSql()) dipakai ketiga call-site -- cegah rumus menyimpang antar tempat. Verifikasi ketiganya kini identik.
  • Dokumentasi asumsi: planned pakai PV level-proyek, bukan bobot per-tugas. Wajar untuk skala TA; tandai sebagai ceiling (ponytail:) bila kelak ingin bobot per-tugas.
  • Konsistensi null: pastikan UI menampilkan "Belum Dinilai" seragam saat spi_value=null.

4.4 Kriteria penerimaan

  • Satu sumber rumus; ketiga endpoint mengembalikan angka identik untuk teknisi yang sama.
  • Teknisi tanpa tugas di proyek aktif terukur -> "Belum Dinilai", bukan 0 atau error.
