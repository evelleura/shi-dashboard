Langkah 2.2 — Query untuk "pamer" data ke penguji

Salin (tombol Salin di pojok kotak), tempel di kotak SQL HeidiSQL, jalankan (F9):

Semua proyek + warna kesehatan + nilai SPI

SELECT p.nama_proyek, p.status, ph.spi_value, ph.status AS warna_rag,
       ph.completed_tasks, ph.total_tasks
FROM tb_proyek p
LEFT JOIN project_health ph ON ph.project_id = p.id_proyek
ORDER BY ph.spi_value ASC;

Tugas suatu proyek (ganti angka 1 dengan id proyek)

SELECT nama_tugas, status, due_date, assigned_to
FROM tb_tugas
WHERE id_proyek = 1
ORDER BY sort_order;

Hitung berapa proyek tiap warna (ringkasan EWS)

SELECT status AS warna, COUNT(*) AS jumlah
FROM project_health
GROUP BY status;

5 Bagaimana EWS (Early Warning System) bekerja untuk dijelaskan

EWS = fitur yang otomatis memberi peringatan kalau proyek mulai tertinggal jadwal, supaya manajer bisa bertindak cepat. Pemicunya: nilai SPI di bawah 1 (progres lebih lambat dari rencana).

Rumus dasarnya (dari naskah BAB II)

SPI = EV / PV

EV (Earned Value / Nilai Hasil)  = % pekerjaan yang BENAR-BENAR selesai
PV (Planned Value / Nilai Rencana) = % pekerjaan yang SEHARUSNYA selesai sampai hari ini
                                   = (Hari Berjalan / Total Durasi Proyek) x 100%
  • SPI = 1 → tepat jadwal → HIJAU
  • SPI > 1 → lebih cepat dari rencana
  • SPI < 1 → terlambat → pemicu peringatan → KUNING / MERAH

Alur kerja di dalam sistem

1. Status tugas berubahteknisi geser / manajer "Selesai"
→
2. Hitung ulang otomatisrecalculateSPI()
→
3. Hitung EV & PVEV = tugas selesai / total
→
4. SPI = EV / PV
5. SPI jadi warnacategorizeHealth()
→
6. Simpan ke project_healthwarna RAG tersimpan

3 cara sistem memberi peringatan

  1. Warna RAG per proyek (merah/kuning/hijau) — lihat sekejap proyek mana yang bermasalah.
  2. Pengurutan kekritisan — dashboard menaruh proyek MERAH paling atas, lalu kuning, lalu hijau. Manajer tidak perlu mencari.
  3. Ringkasan angka di atas dashboard: berapa proyek merah, kuning, hijau, dan berapa lewat tenggat.

7 Kenapa bentuk & struktur code seperti itu alasan desain

Penguji suka bertanya "kenapa dibuat begini?". Tiap keputusan ada alasannya. Hafalkan alasannya saja, bukan code-nya.

1. Memakai Next.js (bukan dua aplikasi terpisah)

Tampilan (frontend) dan logika server (backend) jadi satu proyek. Lebih sederhana dijalankan, sesuai naskah (BAB II 2.2.6 menyebut Next.js + SSR). Semua permintaan lewat satu pintu: api/[...route]/route.ts.

2. Nama tabel Bahasa Indonesia, tapi tampilan tetap normal

Database memakai nama sesuai naskah (tb_proyek, id_proyek). Saat data diambil, namanya "diterjemahkan balik" ke nama umum (id, name) lewat AS di SQL. Alasannya: database setia ke naskah, kode tampilan tidak perlu diubah.

3. Rumus dipisah di file sendiri (spiCalculator.ts)

Rumus SPI/EV/PV terpisah dari tampilan dan database. Alasannya: satu sumber kebenaran — perbaikan cukup di satu tempat; dan mudah diuji.

4. Ada tabel khusus project_health

Daripada menghitung SPI tiap dashboard dibuka (lambat kalau proyek banyak), hasilnya disimpan di project_health dan diperbarui hanya saat ada perubahan tugas. Dashboard tinggal membaca — cepat walau ratusan proyek.

5. "Lewat tenggat" tidak disimpan, tapi dihitung

Status seperti Overtime dihitung langsung dari tanggal tenggat < hari ini, bukan ditandai manual. Alasannya: selalu akurat otomatis, tidak bisa lupa di-update.

6. Teknisi tidak bisa menandai "Selesai" sendiri

Aturan transisi status membatasi teknisi hanya ke "Sedang Dikerjakan". Yang menandai "Selesai" hanya manajer. Alasannya: meniru alur kerja nyata — yang mengerjakan dan yang menyetujui orang berbeda (gerbang review). Menjaga EV tetap jujur.

7. Laporan harian = catatan kendala, BUKAN angka persen

Teknisi tidak mengisi "progres 40%". Sistem menghitung sendiri dari tugas selesai. Laporan harian hanya untuk mencatat kendala. Alasannya: kalau teknisi mengisi angka sendiri, datanya bisa dikarang. Angka yang dihitung sistem lebih objektif.

8. Keamanan

Login pakai token (JWT); tiap permintaan dicek tokennya. Akses dibatasi peran. Semua query ke database memakai parameter ($1, $2), bukan menempel teks — mencegah SQL injection.

Omongin ke penguji (kalimat pamungkas)

"Prinsip desain saya: angka kinerja harus objektif. Karena itu SPI dihitung sistem dari status tugas, bukan diketik manual; hasilnya disimpan agar dashboard cepat; dan ada pemisahan peran supaya data tetap jujur. Struktur kodenya berlapis — tampilan, logika, dan rumus terpisah — agar mudah dirawat dan diuji."

8 Antisipasi pertanyaan penguji + jawaban singkat

"Kenapa SPI tidak diisi manual oleh teknisi?"

Karena angka manual bisa subjektif/dikarang. Sistem menghitung SPI dari jumlah tugas yang benar-benar selesai — lebih objektif, tidak bisa dimanipulasi.

"Kalau proyek belum punya tugas sama sekali, SPI-nya bagaimana?"

Ada cadangan: kalau tidak ada tugas, sistem memakai progres dari catatan laporan harian terakhir; kalau itu pun tidak ada, dianggap 0. (spiCalculator.ts baris 127-136.)

"Apa beda Overtime dan Over Deadline?"

Overtime = tugas sedang dikerjakan tapi sudah lewat tenggat. Over Deadline = tugas belum dikerjakan padahal sudah lewat tenggat. Keduanya dihitung otomatis dari tanggal.

"Kenapa pakai PostgreSQL, bukan MySQL?"

PostgreSQL kuat untuk perhitungan agregat (COUNT berkondisi) yang dipakai dashboard, mendukung tipe data & constraint ketat, andal untuk data proyek relasional.

"Indikator EWS update real-time?"

Nilainya dihitung & disimpan seketika saat status tugas berubah. Tampilan dashboard kadang perlu di-refresh (F5) untuk menampilkan warna terbaru — sudah dicatat sebagai temuan pengujian (TC-MN-04).

"Berapa peran pengguna?"

Dua: Manajer dan Teknisi, sesuai naskah (Tabel 4.9). Manajer kelola & setujui; teknisi kerjakan & lapor.

"Tunjukkan datanya benar ada di database."

Buka HeidiSQL (Bagian 2), jalankan SELECT * FROM tb_proyek; — tampil data mentahnya.

Kalau aplikasi error / blank saat sidang (rencana darurat) 1) Cek terminal masih jalan & Docker Desktop hijau.   2) Refresh browser (F5).   3) Kalau masih kacau, di terminal tekan Ctrl+C, lalu jalankan lagi python run.py --new (tanpa --reset-db supaya data tidak terhapus).   4) Tunggu localhost:3000, buka ulang browser.
